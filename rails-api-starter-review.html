01 — Executive Summary

A foundation that already solves the hard parts

This is one of the more carefully built Rails API starters we've assessed. The code is idiomatic, comments explain why rather than what, and Rails 8 primitives are used correctly (rate_limit, generates_token_for, normalizes, the Solid stack). Most "day-2" operational concerns — health probes, graceful shutdown, structured logging, CI, container hardening — are already handled, which is exactly what most templates skip.

The security model is coherent and tested: JWT access tokens paired with rotating opaque refresh tokens, reuse detection with family revocation, Pundit authorization that defaults to deny, and file-upload validation that runs before the blob is committed. There is no service/form/query layer — and for the current CRUD surface, that restraint is the right call, not a gap.

The five findings that matter most

  • Host authorization fails open in production — an empty ALLOWED_HOSTS accepts any Host header, silently disabling DNS-rebinding protection.
  • Access tokens can't be revoked before their 15-minute expiry — "log out everywhere" only kills refresh tokens. A documented JWT tradeoff that must be stated, not hidden.
  • Login leaks account state & timing — the lockout check runs before the password check, and missing users skip bcrypt, giving two enumeration oracles.
  • Rate limiting is effectively untested — its security spec is skipped, so the most security-relevant control is unverified.
  • No collection filtering / sorting / search — and no allowlist scaffolding, which becomes a scalability and injection-risk gap the moment a second resource lands.
02 — Architecture & Request Flow

How a request actually moves

1 · EdgeRack::Cors → SecureHeaders → Host authorization → param-depth limit (32). /up and /health* bypass the SSL redirect and host checks.
2 · RouterVersioned namespace :api { :v1 }; unknown routes return the JSON envelope, not Rails' HTML 404.
3 · BaseApplicationController adds correlation-id + per-IP rate limit; Api::V1::BaseController mixes in auth, Pundit, and pagination.
4 · Authauthenticate_user! → decode JWT → User.kept.find_by(id:). Public endpoints opt out explicitly.
5 · AuthzPundit policy / scope over a permissions table; verify_authorized & verify_policy_scoped are enforced — a forgotten check is a 403, not a silent data leak.
6 · DataStrong params → Active Record (.kept scope, eager-loaded associations).
7 · OutputPORO serializer (explicit allowlist) → render_success/render_error envelope → JSON. Exceptions unwind through one central handler into the same envelope.
03 — Findings by Priority

What to fix, in order

Critical: none confirmed. No authentication bypass, SQL injection, mass-assignment, cross-tenant leak, or data-loss defect was found — an uncommon and genuine result for a starter of this scope.

High

HighH1Host authorization fails open in production

Host allowlisting is only applied when ALLOWED_HOSTS is set; unset means every Host header is accepted, negating DNS-rebinding and Host-injection protection (poisoned reset links, cache poisoning).

config/environments/production.rb

Fix → Fail loud: raise on boot (or emit a prominent warning) when ALLOWED_HOSTS is blank in production, rather than defaulting to permissive.

HighH2Access tokens are not revocable within their TTL

Logout and password reset revoke refresh tokens only; an already-issued 15-minute access JWT stays valid. Acceptable for many apps, unacceptable for high-security domains.

app/services/jwt_service.rb · Authenticatable#authenticate_user!

Fix → Document the ≤15-min revocation window as an explicit tradeoff; offer an opt-in jti denylist in Solid Cache (TTL = access TTL) rather than building it on by default.

HighH3Login enables account enumeration (state + timing)

The locked? branch returns account_locked before credentials are checked (state oracle), and a non-existent email short-circuits before bcrypt runs (timing oracle) — both undermine the generic failure message.

app/controllers/api/v1/auth_controller.rb

Fix → Check lockout only after a would-be-valid password; run a dummy bcrypt comparison when the user is nil so both paths cost the same.

Medium

MediumM1No filtering / sorting / search, and no allowlist scaffolding

UsersController#index supports pagination + a fixed sort only. Every real API needs query params, and the unsafe implementation (interpolating params[:sort] into order) invites SQL injection — a starter should ship the safe pattern.

Fix → Add a query object / Filterable concern with an explicit allowlist mapping request fields to columns.

MediumM2Account lockout is a self-service DoS vector

Any known email can be locked for 15 minutes with 5 failed attempts, with no IP/device dimension — an attacker can lock out targeted users at will.

Fix → Keep as default but document it; pair with per-IP throttling (already present) and/or exponential backoff instead of a hard lock.

MediumM3Jobs have no explicit retry / discard policy

ApplicationJob relies on Solid Queue defaults — no retry_on/discard_on, no error reporting hook. Future jobs inherit no guardrails.

Fix → Set sensible defaults in ApplicationJob and report failures to the APM.

MediumM4Rate limiting is untested (and fixed-window, DB-backed)

rate_limit_spec.rb is skipped because the null store binds at class load. The limiter is also fixed-window (boundary bursts) and writes to Postgres on every throttled check.

Fix → Test with a real memory store injected in that example group; document that heavy abuse control belongs at the edge (Rack::Attack + Redis, or the LB/gateway).

MediumM5Single-record endpoints serialize without eager loading

me, login, register, and update serialize a user without including roles + avatar, so the serializer fires extra queries. Bounded (one record) so impact is low, but inconsistent with index/show.

Fix → Reload with .includes(:roles, avatar_attachment: :blob), or accept it as negligible and note it.

Low — cleanup

  • Add has_many :refresh_tokens, dependent: :delete_all to User (the FK has no cascade).
  • Password policy is length-only (≥8) — consider an opt-in breach/complexity check.
  • Remove the empty test/ Minitest tree that sits alongside RSpec.
  • JWT has no iss/aud claims — add when tokens cross services.
  • .env.example sets RATE_LIMIT_REQUESTS=10 while code defaults to 300 — reconcile the docs.
  • Index refresh_tokens.expires_at (the purge job filters on it) if the table grows.

05 — Test Coverage

Strong suite, a few real gaps

The security suite is the standout: full refresh-token rotation with reuse → family revocation, per-device vs global logout, auto-unlocking lockout, password-reset session kill, and envelope consistency. Authorization is tested at both policy and request level — including "a non-owner cannot update another user" and default-deny for a role-less user. A real N+1 guard runs in CI via Bullet, and an 80% coverage gate plus an OpenAPI drift check block the build.

Highest-risk untested behaviors

  • Rate limiting — the spec is skipped; the top gap.
  • Mailer rendering — no spec/mailers/; only the enqueue side-effect is asserted.
  • Discarded-user login refusal — logic uses User.kept but is unverified end-to-end.
  • Avatar presign negatives — only the happy path is covered.
  • Confirmation effects — specs assert the envelope, not that confirmed_at is set.
06 — Refactoring Roadmap

A phased path to launch

Phase 1 · Production blockers

Security & data integrity — do these first

  • H1 · Small — Host authorization fails closed / loud.
  • H3 · Small — Fix login enumeration (reorder lock check + dummy bcrypt).
  • H2 · Small–Med — Document the token-revocation window; opt-in jti denylist.
  • M4 · Small — Un-skip and fix the rate-limit test.
Phase 2 · Architecture & consistency

Small correctness and clarity wins

  • Add the correlation ID into the error response body, not just the header.
  • Consistent eager loading on single-record serialization (M5).
  • Remove the empty test/ tree; add the refresh_tokens association.
Phase 3 · Performance & reliability

Ready the template for real resources

  • M1 · Medium — Allowlisted filtering/sorting via a query object.
  • M3 · Small — Job retry/discard + error-reporting defaults.
  • Index refresh_tokens.expires_at for purge at scale.
Phase 4 · Developer experience

Close the coverage and docs gaps

  • Mailer specs, presign/confirmation negatives, discarded-login spec.
  • README: document the token window, lockout DoS, and required LB body-size cap.
  • A multi-tenancy hook/guide — ready before organizations are introduced.
